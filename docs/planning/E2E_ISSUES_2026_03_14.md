## Root Cause Analysis

### N7 [Blocker] — Dashboard "Mock" data after re-auth

**Root cause**: The "Mock" badge is a DEV-only indicator in `PortfolioOverviewContainer.tsx` (line 283):
```
Overview: {hasData ? 'Real' : 'Mock'} | Portfolio: {hasPortfolio ? 'Loaded' : 'None'}
```
`hasData` is `!!resolved.data` from `usePortfolioSummary()`. The summary resolver (`registry.ts` line 351-377) fans out to three parallel calls: `risk-score`, `risk-analysis`, and `performance`. If ANY of those three calls fails, the entire `portfolio-summary` query throws, leaving `data = null` and thus `hasData = false`.

The data flow is: `usePortfolioSummary` -> `useDataSource('portfolio-summary')` -> resolver -> `Promise.all([risk-score, risk-analysis, performance])`. After re-auth, the portfolio IS loaded in the store (`hasPortfolio = true`), but the three sub-queries may fail due to:
1. **Stale TanStack Query cache from before logout** not being invalidated on re-auth. The `createAuthStore.ts` `clear()` (line 86-98) resets auth state but does NOT call `queryClient.invalidateQueries()`. On re-auth, stale cached errors from previous session may persist.
2. **The `onUnauthInit` callback** (line 135) is called when init finds no session, but there's no equivalent cache clear on successful re-auth via `initializeAuth`.
3. **Plaid 500 (N9)** could cascade: if the Plaid refresh fails during portfolio bootstrap in `PortfolioInitializer` (line 204-221), it's caught and logged, but the portfolio_data may have stale holdings that cause downstream `risk-analysis` or `risk-score` to fail.

**Key files**:
- `frontend/packages/ui/src/components/dashboard/views/modern/PortfolioOverviewContainer.tsx` lines 283, 69-75
- `frontend/packages/connectors/src/resolver/registry.ts` lines 351-377 (portfolio-summary resolver)
- `frontend/packages/connectors/src/features/portfolio/hooks/usePortfolioSummary.ts`
- `frontend/packages/app-platform/src/auth/createAuthStore.ts` lines 116-148 (initializeAuth)

### N10 [Major] — Portfolio selector click triggers full page reload

**Root cause**: The PortfolioSelector `handleSelect` (line 134-170) does NOT use `window.location` — it uses `PortfolioRepository.add()` + `PortfolioRepository.setCurrent()` + `queryClient.invalidateQueries()`, which are all client-side state changes. No page reload should occur from this code path.

The probable cause is the **`queryClient.invalidateQueries()` call (line 166) with no filter** — this invalidates ALL queries including the `initialPortfolioKey` query. `PortfolioInitializer` has `enabled: !currentPortfolio` (line 142-146), and the rapid state transition during `setCurrent` could momentarily set `currentPortfolioId` to a new value, then `invalidateQueries()` re-triggers queries that were already settled, causing a cascade. However, since `setCurrent` is synchronous (Zustand `setState`), this shouldn't cause a full page navigation.

The full page reload symptom ("Checking authentication..." hang) suggests the entire React tree unmounts and re-mounts, which means the auth store's `isInitialized` flag resets. This could happen if:
1. **The `DropdownMenuItem.onSelect` event** propagates and triggers a native link/form behavior. The selector uses `event.preventDefault()` (line 245, 248), so this should be blocked.
2. **A crash in the query invalidation cascade** causes the React error boundary to unmount the tree. The `DashboardErrorBoundary` might not catch all errors.
3. **The `SessionServicesProvider` unmounts and remounts** due to the auth state briefly flickering during heavy query invalidation, causing all session services to reset.

The most likely culprit is that `queryClient.invalidateQueries()` with no arguments triggers a massive refetch storm, some of which fail (especially if session has expired — N13), causing cascading errors that crash the React tree.

**Key files**:
- `frontend/packages/ui/src/components/dashboard/PortfolioSelector.tsx` lines 134-170 (handleSelect)
- `frontend/packages/connectors/src/providers/PortfolioInitializer.tsx` lines 140-146
- `frontend/packages/chassis/src/repository/PortfolioRepository.ts` line 331 (setCurrent)

### N13 [Major] — No session expiration recovery UX

**Root cause**: **HttpClient has NO 401 interceptor.** The `fetchWithRetry` method (line 67-99 in `HttpClient.ts`) only special-cases status 429 (rate limit). For status 401, it creates a generic `RetryableHttpError` (line 80: `throw this.createHttpError(response)`) which is then retried up to 3 times with exponential backoff (line 94). After 3 failed retries (all returning 401), the error is thrown to the caller.

The auth store (`createAuthStore.ts`) has no mechanism to detect 401 errors from API calls and transition to unauthenticated state. The `signOut()` method (line 76) must be called explicitly. There is no global interceptor that listens for 401 responses and triggers `signOut()` or shows a session-expired banner.

When the InMemorySessionStore session expires:
1. API calls start returning 401
2. HttpClient retries each 3 times (wasting ~7 seconds per request: 1s + 2s + 4s backoff)
3. TanStack Query receives the error, enters error state per-query
4. Some queries have cached data (React Query `staleTime`), so those components still show stale metrics
5. Queries without cached data show error states
6. The user sees a mixed state: some cards with cached data, others broken
7. No mechanism detects this pattern and shows a "session expired" banner

**Key files**:
- `frontend/packages/app-platform/src/http/HttpClient.ts` lines 67-99 (fetchWithRetry — no 401 handling)
- `frontend/packages/app-platform/src/auth/createAuthStore.ts` (no 401 detection)

### N2 [Major] — Holdings empty for single-account portfolio

**Root cause**: The Holdings view uses `usePositions()` -> `useDataSource('positions-enriched')` -> resolver `registry.ts` line 158-198. For non-manual portfolios, the resolver calls:
```typescript
const portfolioName = getPortfolioName(undefined, portfolio);
const payload = await api.getPositionsHoldings(portfolioName);
```
`getPortfolioName` (line 84-92) extracts `portfolio_name` from the portfolio object. If the portfolio in the store was loaded with the `_auto_*` name, it sends that raw name to `GET /api/positions/holdings?portfolio_name=_auto_interactive_brokers_...`.

The backend `_load_enriched_positions()` in `routes/positions.py` calls `resolve_portfolio_scope()` which may not recognize the `_auto_*` name and returns empty positions. The Dashboard's overview works differently — it uses `risk-analysis` and `risk-score` which go through `SessionManager` using `portfolio.id` (the Zustand store key) rather than `portfolio_name`, and those managers resolve scope via the backend's portfolio config path which may handle the name differently.

The core issue is that the `_auto_*` portfolio name is not being resolved to the correct account scope when passed to the enriched positions endpoint.

**Key files**:
- `frontend/packages/connectors/src/resolver/registry.ts` lines 158-198 (positions-enriched resolver)
- `frontend/packages/connectors/src/resolver/registry.ts` lines 84-92 (getPortfolioName)
- Backend: `routes/positions.py` (`_load_enriched_positions`)
- Backend: `services/portfolio_scope.py` (`resolve_portfolio_scope`)

### N11 [Major] — Alerts cross-scope contamination

**Root cause**: The alerts endpoint `GET /api/positions/alerts` (in `routes/positions.py` line 410-476) does NOT accept a `portfolio_name` parameter. It always calls:
```python
service = PositionService(user_email=user["email"], user_id=user["user_id"])
result = service.get_all_positions(consolidate=True)
```
This loads ALL positions across all accounts (the "combined" view) and generates alerts from that combined set. There is no portfolio scope filtering.

On the frontend, `useSmartAlerts()` calls `useDataSource('smart-alerts')` which resolves to `api.getPortfolioAlerts()` (registry.ts line 662-667). The resolver does NOT pass any portfolio context — it calls the endpoint with no parameters. The TanStack Query key for `smart-alerts` is `['sdk', 'smart-alerts']` which is the same regardless of selected portfolio, so portfolio switches don't invalidate this cache either.

The `DashboardAlertsPanel` additionally derives alerts from `useRiskScore()` component_scores (line 75-92), which IS portfolio-scoped. So the panel shows a mix of globally-scoped smart alerts (from combined positions) and correctly-scoped risk score alerts.

**Key files**:

### N12 [Major] — Position count mismatch (21 vs 15)

**Root cause**: Two different data sources count positions differently:

1. **"21 positions"** in PortfolioSelector comes from `portfolio.position_count` in the portfolio list API response. The backend `_build_portfolio_payload` (in `mcp_tools/portfolio_management.py` lines 112-117) counts ALL positions from `db_client.get_portfolio_positions()` or `filter_positions_to_accounts()`. This includes options, cash entries, and any other instrument types stored in the positions table.

2. **"View All 15 Holdings"** in DashboardHoldingsCard comes from `holdings.length` (line 158), where `holdings = positionsData?.holdings ?? []`. This data comes from `usePositions()` -> `positions-enriched` resolver -> `api.getPositionsHoldings()` -> `PositionsAdapter.transform()`. The backend enriched positions endpoint may filter differently (e.g., excluding options, zero-quantity positions, or cash equivalents). The `PositionsAdapter.transform()` (line 101-117) maps ALL positions from the response without filtering, so the difference is in what the backend returns.

The two counts use different backend data sources: one is the raw DB position count, the other is the filtered/enriched positions monitor pipeline.

**Key files**:
- `mcp_tools/portfolio_management.py` lines 112-117 (`_build_portfolio_payload` position_count)
- `frontend/packages/ui/src/components/dashboard/cards/DashboardHoldingsCard.tsx` line 158

### N16 [Major] — Concentration "100 / Well Diversified" contradiction

**Root cause**: The concentration score comes from the risk score API's `component_scores.concentration_risk` field. Per the `RiskScoreAdapter` documentation (lines 44-48), the backend returns:
```
concentration_risk: number (0-100)  // 100 = best (well diversified), 0 = worst (concentrated)
```
The `PortfolioSummaryAdapter` (line 380-383) extracts this as the "Concentration Risk" component score. The frontend label mapping in `useOverviewMetrics.ts` (line 96) is:
```
score >= 70 ? "Well Diversified" : score >= 40 ? "Moderate" : "Concentrated"
```
This mapping is CORRECT for the score semantics (high = diversified).

The bug is that the **backend concentration_risk score = 100 is wrong for the Combined portfolio**. If AAPL=25.7% and MSFT=30.0% (combined >55% in two names), the concentration score should NOT be 100. The backend risk score computation for the Combined portfolio is returning an incorrect maximum score. This could be because:
1. The Combined portfolio risk score is being computed with incorrect weights (e.g., not reflecting the actual combined positions)
2. The concentration risk scoring formula has a bug for larger portfolios where individual HHI drops below thresholds despite having concentrated top holdings
3. The risk limits used for the Combined portfolio may have very permissive concentration thresholds

The label mapping itself is correct — the score from the backend is the issue.

**Key files**:
- `frontend/packages/ui/src/components/portfolio/overview/useOverviewMetrics.ts` line 96 (label mapping — correct)
- `frontend/packages/connectors/src/adapters/PortfolioSummaryAdapter.ts` lines 380-383
- Backend: risk score computation for Combined portfolio (services/portfolio_service.py `analyze_risk_score`)
- Config: `risk_limits.yaml` concentration thresholds

### N5 [Minor] — Trading analysis HTTP 500

**Root cause**: The literal string `CURRENT_PORTFOLIO` is being sent as the portfolio_name. In `registry.ts` line 91, the `getPortfolioName()` function falls back to `'CURRENT_PORTFOLIO'` when the portfolio object has no `portfolio_name` field:
```typescript
return currentName || 'CURRENT_PORTFOLIO';
```
The `trading-analysis` resolver (line 380-441) calls `getPortfolioName(undefined, portfolio)` to get the name, then passes it to `api.getTradingAnalysis(portfolioName)`. If the portfolio in the store lacks `portfolio_name` (or it's empty string), the literal string `CURRENT_PORTFOLIO` is sent.

The backend trading analysis endpoint likely doesn't resolve `CURRENT_PORTFOLIO` as a valid portfolio scope, causing a 500. The frontend handles this gracefully in the resolver (lines 383-395) by catching the error and returning `null` for `backendData`.

### N4 [Minor] — 7x setState-during-render warnings

**Root cause**: All 7 components share a common pattern: they use `useDataSource()` which internally calls `useSyncExternalStore` (line 75-81 in `useDataSource.ts`) to track TanStack Query cache changes:
```typescript
const dependencyVersion = useSyncExternalStore(
    (onStoreChange) => queryClient.getQueryCache().subscribe(onStoreChange),
    () => queryClient.getQueryCache().findAll({ queryKey: ['sdk'] })
      .map(...).join('|'),
    () => ''
);
```
When TanStack Query updates its cache (e.g., a query resolves), the `subscribe` callback fires `onStoreChange`, which triggers React to re-render the subscribing component. If that re-render happens while a DIFFERENT component is already rendering (the "Cannot update a component while rendering a different component" warning), it triggers the React warning.

The specific pattern is: during the initial Dashboard load, multiple `useDataSource` hooks across 7 containers all subscribe to the query cache. When one query resolves, the cache update notification fires `useSyncExternalStore`'s `onStoreChange` callback in all subscribers, which calls `setState` on those components during the render cycle of the component that triggered the cache update.

This is a known React 18 strict-mode/concurrent pattern issue with `useSyncExternalStore` subscriptions to shared external stores. It's cosmetic (does not affect correctness) but noisy.

**Key files**:
- `frontend/packages/connectors/src/resolver/useDataSource.ts` lines 75-81 (useSyncExternalStore subscription)
- All 7 components listed in the issue (they all use hooks that call `useDataSource` internally)

### N3 [Minor] — Settings risk management stuck loading

If `useRiskScore()` returns an error, `riskScoreData` is undefined, which is passed to `riskSettingsAdapter.transform(riskSettingsResult.riskSettings, riskScoreData)` — the adapter may handle this gracefully, but the issue could also be that the `riskSettingsManager.getRiskSettings()` call itself hangs for the single-account scope.

### N14 [Minor] — Total Allocation shows "100.1%"

**Root cause**: `AssetAllocation.tsx` line 183 computes:
```typescript
const totalAllocation = allocations.reduce((sum, item) => sum + (Number(item.percentage) || 0), 0);
```
And displays it at line 521 with `formatPct(totalAllocation)` which is `value.toFixed(1) + '%'`.

The backend `risk-analysis` endpoint returns per-asset-class percentages that are individually rounded but not constrained to sum to exactly 100.0. Each percentage is a floating-point value, and the sum of individually-rounded floats can exceed 100.0 (e.g., 60.05 + 25.02 + 15.03 = 100.1).

**Key files**:
- `frontend/packages/ui/src/components/portfolio/AssetAllocation.tsx` lines 183, 521

#### Fix 4a: Replace useSyncExternalStore with useEffect for dependency tracking [N4]
**Files**: `frontend/packages/connectors/src/resolver/useDataSource.ts`

Replace the `useSyncExternalStore` subscription to the TanStack Query cache (lines 75-81) with a `useEffect` that re-evaluates dependency readiness after cache updates:

```typescript
const [dependencyVersion, setDependencyVersion] = useState('');
useEffect(() => {
    return queryClient.getQueryCache().subscribe(() => {
        const newVersion = queryClient.getQueryCache().findAll({ queryKey: ['sdk'] })
            .map((q) => `${q.queryHash}:${q.state.dataUpdatedAt}:${q.state.status}`)
```

This moves the state update out of the render cycle and into an effect, eliminating the "Cannot update during render" warning.
