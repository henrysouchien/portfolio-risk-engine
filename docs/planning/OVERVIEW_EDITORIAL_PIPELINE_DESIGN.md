## Problem Statement

The Overview is the analyst's morning briefing, but it currently shows every user the same thing: 6 hardcoded metrics (Total Value, YTD Return, Max Drawdown, Risk Score, Sharpe, Alpha) with insight text selected by a mechanical severity cascade (error > warning > info > success). No awareness of who the user is. No editorial judgment.

The product has 80+ tools computing everything from factor exposure to tax harvest opportunities. The data is there. The gap is editorial selection — choosing what to lead with, what to emphasize, what to omit based on who this specific person is and what matters to them right now.

This is NOT a visual design problem (the design system handles that). This is an information architecture problem: what does the analyst choose to tell you, and why?

## Demand Evidence

- **Founder is user #1**: Built the product for personal use. Knows from equity research experience that the right "lead story" changes based on what the investor cares about and what's happening in the portfolio.
- **Diverse first cohort**: Education program students include retail investors (want P&L tracking), active traders (TradingView users wanting momentum/entries/exits), and an institutional analyst (FactSet/Bloomberg background, exploring modern tools).
- **Specific demand signal**: One student (former Anton Kriel course) specifically asked for position + P&L tracking and is willing to pay.
- **Design conviction**: A one-size-fits-all briefing won't work for different investor types. A dividend investor needs yield and payment dates first, a growth investor needs concentration and earnings, a risk-conscious user needs drawdown and factor exposure.

## Status Quo

The cohort currently uses a fragmented workflow:
- **Broker apps** (IBKR, Schwab) for positions — functional but overwhelming or too limited
- **Koyfin** (founder recommends) for portfolio analytics
- **TradingView** for trading-focused users
- **FactSet/Bloomberg/Capital IQ** for institutional analyst (exploring alternatives, using Perplexity)
- **Yahoo Finance** and spreadsheets as fallbacks

No tool in this stack provides a personalized, opinionated morning briefing. They all show the same dashboard to every user. The analyst model — where someone who knows YOU picks what you need to see — doesn't exist at the retail level.

## Target User & Narrowest Wedge

**Target user:** The founder, as user #1. Students follow because they signed up to learn his investment approach. This is not a broad consumer-personalization problem yet — it's an expert-led editorial product for a specific investing archetype.

**Narrowest wedge:** Replace the hardcoded Overview with an editorially curated briefing that feels "picked for me." Same layout, different content selection per user. Prove the editorial model works with 1 user (founder) and ~10-20 beta users.

## Constraints

1. **Layout is fixed** — the design system defines the visual structure (metric strip, lead insight, sections). The AI is the editor-in-chief, not the page designer.
2. **No settings UI** — no preferences page, no configuration form. The profile is built through conversation and observation, not a settings panel. The AI maintains the `editorial_memory`, not the user.
3. **Design system implementation is in flight** — visual concerns are being handled in a separate worktree (`codex/big-impl`). This design is orthogonal to the visual rollout.
4. **Existing agent format** — 16+ tools return `{status, format: "agent", snapshot, flags}`. Insight candidates come from this pipeline.
5. **No external users yet** — curated beta is the next milestone. Personalization must prove its value on the founder first.

## Premises

1. **The AI makes editorial judgments informed by a persistent `editorial_memory`** — a structured object it reads and writes through conversation, not a user-managed UI. Captures goals, style, time horizon, current focus, sophistication level. (Revised per Codex challenge — "if the AI remembers, that IS a profile. Make it explicit and reliable, not implicit and sloppy.")
2. **The Overview is a structured layout with editorial discretion** — the AI picks what fills each slot. Editor-in-chief, not page designer.
3. **Nail the default for the founder first** — students follow, so one great "house view" is the starting point.
4. **Data is all there, the gap is editorial selection** — the 80+ tools compute everything. The problem is choosing what to lead with, emphasize, or omit.

## Approaches Considered

### Approach A: Slot Composer (Minimal Viable)

Single `compose_overview()` function. Gathers data from existing tools, scores candidates via deterministic ranking, LLM picks final slots. editorial_memory as a single JSON blob.

- Effort: M (~1-2 weeks)
- Risk: Low
- Pros: Ships fastest, minimal new abstractions
- Cons: Ranking logic in one function, no observability, harder to extend

### Approach B: Editorial Pipeline (Ideal Architecture) — CHOSEN

Modular insight generators (concentration, risk, performance, income, trading, factor, tax, events) each produce scored candidates with structured metadata. Editorial policy layer combines scores + editorial_memory. LLM final arbiter picks from top N candidates (async, non-blocking). Each slot gets "Why you're seeing this" metadata. Generators are pluggable. Phased rollout: 4 core generators first, expand after validation. Engagement logging deferred to Phase 2.

- Effort: L (~3-4 weeks)
- Risk: Medium
- Pros: Observable, extensible, each generator testable independently, logging built in, matches codebase patterns (three-layer architecture)
- Cons: More infrastructure before learning defaults

### Approach C: Morning Brief Artifact (Creative/Lateral)

The Overview IS an AI-generated document via :::ui-blocks. Generated on a cadence, cached for the day.

- Effort: M (~2 weeks)
- Risk: High (LLM latency, unpredictable output)
- Pros: Most flexible, closest to "real analyst"
- Cons: Slow, hard to cache, goes against "define the structure for the AI" instinct

## Recommended Approach

**Approach B: Editorial Pipeline.** Matches the codebase's established patterns (three-layer agent format, provider protocols, pluggable generators). The founder builds infrastructure first, not quick hacks — this is consistent with 80+ tools, 3000+ tests, and a CLAUDE.md that says "NO COWBOY CODING."

### Phased Generator Rollout

**Phase 1 (prove the model):** 4 core generators — Concentration, Risk, Performance, Events. These cover the founder's primary concerns and the most common "lead stories." Validate that the editorial pipeline produces better lead insights than the hardcoded severity cascade.

**Phase 2 (expand coverage):** Add Income, Trading, Factor, Tax Harvest generators after Phase 1 proves the editorial model works. Each generator drops in without touching the composer.

### AI-Generated Artifacts as Editorial Output

The editorial pipeline has two modes of output, not one:

1. **Selection** — pick what goes in fixed slots (metric strip, lead insight, attention items). Deterministic, fast, cacheable. This is the core pipeline.

2. **Generation** — create custom artifacts for this specific portfolio state. A one-off chart, a narrative section, a visual breakdown that no pre-built component can express. LLM-driven via the existing `:::ui-blocks` artifact system.

The Overview already integrates AI-generated artifacts. The editorial layer sits on top: the AI decides WHAT artifacts to generate based on editorial_memory + portfolio state, not just which pre-built components to fill. The design system (DESIGN.md) constrains the visual language — the AI generates within those rules, not freeform.

This extends to the **chat margin** (the annotation layer from DESIGN.md), which is already designed as an AI surface. The editorial pipeline informs what appears there too — "Ask the analyst about this" prompts, revision marks on changed insights, contextual annotations. The margin is where the analyst's ongoing commentary lives alongside the structured briefing.

Three AI surfaces on the Overview, all editorially controlled:
- **Structured slots** (metric strip, lead insight, attention items) — selection from generators
- **Generated artifacts** (custom charts, narratives, breakdowns) — LLM creates when the situation warrants
- **Chat margin** (annotations, "ask about this", revision marks) — AI commentary layer alongside the briefing

### Two-Speed Architecture: Deterministic Fast Path + Async LLM Enhancement

The pipeline has two speeds to eliminate LLM latency from the critical render path:

1. **Deterministic fast path (synchronous, <200ms for pipeline logic after data is gathered):** Generators + editorial policy scoring. Produces a complete `OverviewBrief` with ranked candidates in each slot. Data gathering latency depends on upstream tool cache state (piggybacks on results already fetched by other views like `usePortfolioSummary`/`usePerformance`; cold calls may take 2-5s for risk engine computation). Metric labels use template strings, lead insight uses the top candidate's `why` field. This renders immediately.

2. **LLM enhancement (asynchronous):** Runs in background after fast path renders. Rewrites metric labels with context, composes the lead insight in analyst voice, reorders attention items by narrative flow. Frontend swaps in enhanced brief when ready (React query invalidation). If the LLM fails or times out, the deterministic brief stands.

**Caching:** Brief cached for 1 hour per user. Invalidated on: portfolio data refresh, new transaction, or explicit "refresh" action. editorial_memory changes trigger cache invalidation. The LLM-enhanced version replaces the deterministic version in cache once generated.

### Onboarding Conversation Flow

The analyst interviews the user through the existing chat system. Not a form. Not a wizard. A conversation.

**Opening:** "I'm your portfolio analyst. Before I start working on your briefing, I want to understand how you think about investing. What matters most to you when you look at your portfolio?"

**Follow-ups based on response:**
- If they mention risk → "What kind of risk keeps you up at night — concentration in a single name, sector exposure, or drawdown from an overall market event?"
- If they mention returns → "Are you tracking against a benchmark, or is it more about absolute return and income generation?"
- If they mention trading → "Are you actively trading, or more buy-and-hold with occasional rebalancing?"
- If they say "I don't know" → "That's fine — I'll start with a balanced briefing and adjust as I learn what you find useful."

**The analyst updates editorial_memory after the conversation and after observing what the user clicks, asks about, or ignores over time.**

### Insight Generator Interface

```python
@dataclass
class InsightCandidate:
    slot_type: Literal["metric", "lead_insight", "attention_item"]
    content: dict          # display text, value, context
    relevance_score: float # 0-1, how relevant to this portfolio right now
    urgency_score: float   # 0-1, is this time-sensitive?
    novelty_score: float   # 0-1, based on (category, slot_type) presence in previous brief + evidence_hash for value-change detection within same category
    category: str          # concentration, risk, performance, income, trading, factor, tax, events
    evidence: list[str]    # data points backing this insight
    why: str               # human-readable "why you're seeing this"
    source_tool: str       # which MCP tool generated the underlying data

class InsightGenerator(Protocol):
    def generate(self, portfolio_snapshot: dict, flags: list[dict]) -> list[InsightCandidate]:
        """Produce insight candidates from tool output."""
        ...
```

Each generator wraps an existing tool's agent-format output. The generator's job is to read the snapshot + flags and produce scored InsightCandidates. The scoring is deterministic — no LLM in the generators, just data-driven relevance/urgency/novelty computation.

### What Changes on the Frontend

Minimal. The Overview layout stays the same (metric strip, lead insight, attention items). What changes:

1. **Data source**: Instead of `useOverviewMetrics()` with hardcoded metric definitions, the frontend consumes the `OverviewBrief` payload from a new `GET /api/overview/brief?portfolio_id={id}` endpoint (auth via existing session middleware, returns `OverviewBrief` JSON).
2. **Metric labels**: Currently static ("Total Portfolio Value"). With the brief, labels can be contextualized ("Portfolio Value — up $4,200 driven by AAPL earnings beat").
3. **Lead insight**: Currently from the highest-severity flag. Now from the LLM's editorial selection.
4. **Attention items**: Currently AI Recommendations from factor intelligence only. Now from any generator — trading signals, tax harvest deadlines, upcoming events, risk warnings — whatever the pipeline ranks highest.
5. **"Why you're seeing this"**: Optional expandable on each slot for transparency.

## Resolved Design Decisions

1. **Caching:** Brief cached 1 hour per user. Invalidated on portfolio refresh, new transaction, or editorial_memory change. Deterministic fast path is the synchronous render; LLM enhancement updates cache async.
2. **Fallback:** Deterministic brief IS the default. LLM enhancement is a quality upgrade, not a requirement. If LLM fails/times out, the deterministic brief stands with no degradation.
3. **Observation loop:** Deferred to Phase 2. At <20 users, implicit engagement signals are too noisy to be useful. Phase 1 relies on explicit editorial_memory from conversation + the founder's manual briefing exercise.

## Open Questions

1. **editorial_memory storage**: DB column on user table? Separate key-value store? File-based per user? (Recommend: JSONB column on user table — simplest, already have the schema.)
2. **Generator activation**: All Phase 1 generators run on every load (only 4 tools, parallel). Phase 2 may introduce selective activation based on editorial_memory.
3. **Relationship to Scenario Workflows**: The existing "AI Recommendations" section feeds from factor intelligence only. Does it get absorbed into the pipeline, or does it stay separate?
4. **Previous brief storage**: Novelty scoring requires knowing "was this true in the last brief?" Need to persist the previous `OverviewBrief` per user. (Recommend: same DB table as editorial_memory, simple JSON column.)
5. **editorial_memory write path (Phase 2, NEEDS DESIGN)**: Chat history is client-owned — there is no server-side conversation persistence. The onboarding conversation can't automatically update editorial_memory without a dedicated write mechanism. Key questions: (a) What triggers a memory write — does the AI decide, the user confirm, or is it automatic after onboarding? (b) How does the AI extract structured editorial_memory fields from freeform conversation? (c) How do corrections work ("actually I don't care about dividends anymore")? (d) Retention/cleanup — does conversation_extracts grow unbounded? (e) Privacy — what gets stored, what doesn't? Phase 1 sidesteps this entirely with a hardcoded editorial_memory. But Phase 2's onboarding conversation depends on solving this. (Flagged by Codex outside voice review.)

## Success Criteria

1. **The founder opens the Overview and the lead insight is something they care about** — not a mechanical severity cascade, but an editorially chosen story.
2. **Two different users with different editorial_memories see different lead stories** for the same portfolio state.
3. **"Why you're seeing this" is coherent** — the system can explain its editorial choices.
4. **No regression in load time** — the pipeline should cache aggressively and the brief should feel instant.
5. **The onboarding conversation takes < 2 minutes** and produces a usable editorial_memory.
