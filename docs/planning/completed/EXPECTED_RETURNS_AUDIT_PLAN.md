## 1. Problem Statement

`config/portfolio.yaml` contains 16 hardcoded `expected_returns` values that were manually entered at some unknown point in the past. At least one value is severely stale: **SFM shows 0.438 (43.8%)** while the pipeline correctly computes ~7.5% via its industry proxy XLP (10yr CAGR). The pipeline (`ReturnsCalculator.estimate_historical_returns`) works correctly -- the problem is that stale YAML values are read directly and never refreshed.

### Root cause (unverified — most likely explanation)

The `expected_returns` block was introduced in commit `c73b2866` (March 14, 2026 — "accumulated deployment sprint"). That commit is a large squash merge that includes `config/portfolio.yaml` as a new file. The `ReturnsCalculator` already used the industry ETF CAGR method in its very first committed version (also `67501198`), and it contained a fallback path: if a ticker was not present in `stock_factor_proxies`, the calculator fell back to using the stock's own CAGR (`"Not in stock_factor_proxies, using own CAGR"`).

**However**, `git show c73b2866:config/portfolio.yaml` shows that all 16 tickers already had industry proxy entries in `stock_factor_proxies` at that commit (NVDA→SOXX, SFM→XLP, RNMBY→ITA, etc.). So the missing-proxy fallback does NOT explain the inflated values for the committed YAML — the proxies were present.

**Possible explanations** (none definitively verified):

1. **Pre-squash state**: Commit `c73b2866` is a squash merge. The `expected_returns` values may have been generated in an earlier working-tree state where `stock_factor_proxies` was incomplete or absent, then committed alongside a later version of the proxy config. The squash collapsed this history.
2. **Manual entry or external script**: The values may have been generated by a different script or entered manually, not by `ReturnsCalculator` at all. The floating-point noise (e.g., `0.29100000000000004`) suggests programmatic generation but does not prove which code path produced them.
3. **Different calculator version**: A pre-commit version of the calculator may have used stock CAGR as the primary method before the industry-ETF-first logic was written, but that version was never committed.

**What IS verified**: The current pipeline correctly uses industry ETF CAGR. The problem is that `ensure_returns_coverage()` only fills in **missing** tickers — it never overwrites existing stale values. Regardless of how the stale values got there, the fix is the same.

## 3. Fix Approach

### Phase 1: One-time data fix (immediate)

**Regenerate all expected returns values using the pipeline.**

1. Run `ReturnsCalculator.estimate_historical_returns()` for all non-cash-proxy portfolio tickers
2. Compare pipeline values against stored YAML values
3. Write corrected values to `config/portfolio.yaml` via `update_portfolio_expected_returns(replace_all=True)`
4. Also regenerate and save to the database for the production user

**SGOV cash proxy exclusion**: SGOV (and IBGE.L, ERNS.L) are cash proxies listed in `config/cash_map.yaml` under `proxy_by_currency`. They must be **excluded** from the bulk refresh because `get_complete_returns()` has a precedence bug: stored/temp returns (line 404) beat the Treasury-rate path (line 411-424). If the refresh script writes `SGOV=0.123` (its own ETF CAGR) to the YAML or DB, `get_complete_returns()` will return `0.123` for SGOV instead of the correct Treasury rate (`~0.051`). The cash proxy Treasury-rate path only fires when no stored/temp return exists. The fix: exclude cash proxy tickers from bulk refresh entirely, and remove any existing SGOV entry from `expected_returns` in YAML/DB so the Treasury-rate path is always reached.

This is a manual/scripted operation. The output should be logged for audit trail.

**Script** (to be run once, not committed):
```python
from inputs.returns_calculator import ReturnsCalculator
import yaml
from config import resolve_config_path

rc = ReturnsCalculator()

# Load cash proxy tickers to EXCLUDE from bulk refresh
with open(resolve_config_path("cash_map.yaml")) as f:
    cash_map = yaml.safe_load(f)
cash_proxy_tickers = set(cash_map.get("proxy_by_currency", {}).values())
# => {'SGOV', 'IBGE.L', 'ERNS.L'}

all_tickers = ["SGOV", "DSU", "EQT", "IGIC", "IT", "KINS", "MSCI", "NVDA",
               "RNMBY", "SFM", "SLV", "STWD", "TKO", "V", "AAPL", "MSFT"]

# Only refresh non-cash-proxy tickers
refresh_tickers = [t for t in all_tickers if t not in cash_proxy_tickers]
skip_tickers = [t for t in all_tickers if t in cash_proxy_tickers]
print(f"Refreshing: {refresh_tickers}")
print(f"Skipping (cash proxies, handled by Treasury-rate path): {skip_tickers}")

fresh = rc.estimate_historical_returns(refresh_tickers, years_lookback=10)

# Compare
with open(resolve_config_path("portfolio.yaml")) as f:
    stale = yaml.safe_load(f).get("expected_returns", {})

for t in all_tickers:
    old = stale.get(t, "MISSING")
    if t in cash_proxy_tickers:
        print(f"{t:8s}  old={old}  SKIP (cash proxy — uses Treasury rate)")
    else:
        new = fresh.get(t, "MISSING")
        delta = ""
        if isinstance(old, float) and isinstance(new, float):
            delta = f"  delta={new - old:+.1%}"
        print(f"{t:8s}  old={old}  new={new}{delta}")

# Write fresh values (cash proxies excluded — their YAML entries should be REMOVED)
rc.update_portfolio_expected_returns(fresh, replace_all=True)
# NOTE: This replace_all will drop SGOV from expected_returns since it's not in `fresh`.
# That is the desired behavior — SGOV should fall through to the Treasury-rate path.
```

### Phase 2: Staleness guard (code change)

Add a validation check that warns when stored expected returns deviate significantly from what the pipeline would compute. This prevents the problem from recurring.

#### 2a. `data_source` tagging — scope correction

**Problem with original plan**: The plan said `ensure_returns_coverage()` should pass `data_source='calculated'` through the save path. But `ensure_returns_coverage()` (line 307-320) explicitly stores generated returns **in memory only** (`self._temp_generated_returns`) and does NOT persist them to the database. The comment at line 317 says "Store generated returns in memory (NOT in database)". Adding a `data_source` parameter to `DatabaseClient.save_expected_returns()` alone would have no effect because the auto-gen path never calls it.

**Revised approach**: There are two options:

**Option A (recommended): Leave auto-gen as temp-only, add `data_source` to the manual save path only.** The `save_expected_returns()` in `database_client.py` (line 1761) currently hardcodes `data_source='user_input'`. Accept a `data_source` parameter (default `'user_input'`) so callers like the Phase 1 refresh script and `audit_expected_returns()` can tag rows as `'calculated'` when they persist corrected values. Auto-generated temp returns remain transient and do not need tagging.

**Option B (deferred): Wire persistence into `ensure_returns_coverage()`.** This would require `ensure_returns_coverage()` to call `save_expected_returns()` with `data_source='calculated'` instead of only storing in `_temp_generated_returns`. This is a larger design change because temp-only was an intentional choice — auto-generated values should not silently become permanent without user review. Defer this.

For the YAML path, metadata tagging is not worth adding (YAML is fallback/dev mode only).

#### 2b. Add pipeline-vs-stored deviation audit to `ReturnsService`

Add a standalone `audit_expected_returns()` method to `ReturnsService`. This is NOT added to `ValidationService` because `ValidationService` is a pure structural validator (type checks, NaN/inf guards) with no service dependencies — it takes `PortfolioData` and checks data shapes. Staleness auditing requires `ReturnsCalculator` + FMP API calls, which belong in `ReturnsService`.

The audit is expensive (FMP calls per ticker) so it should be:

- **Never called in hot paths** (API requests, optimizer calls)
- Triggered explicitly via an `audit_expected_returns` function or MCP tool
- Optimization pre-flight wiring is deferred (see Section 2d)

**Implementation**: Add a new method `audit_expected_returns()` to `ReturnsService`:

```python
def audit_expected_returns(
    self,
    portfolio_name: str,
    deviation_threshold: float = 0.15,  # 15pp absolute deviation triggers warning
    years_lookback: int = None,
) -> Dict[str, Any]:
    """
    Compare stored expected returns against pipeline-computed values.
    
    Returns dict with:
        - stale_tickers: list of tickers where |stored - pipeline| > threshold
        - audit_details: per-ticker comparison
        - recommendation: "refresh_all" | "refresh_stale" | "ok"
    """
```

This method:
1. Loads stored expected returns
2. Runs `estimate_historical_returns()` for all portfolio tickers
3. Compares each stored value against pipeline value
4. Returns a structured audit result

**Threshold choice**: 15 percentage points absolute deviation. This catches the obvious errors (SFM: 43.8% stored vs 7.5% pipeline = 36.3pp deviation) while allowing reasonable manual overrides (e.g., user sets AAPL to 15% vs pipeline 18% = 3pp, within tolerance).

#### 2c. Fix DB 0.0 truthiness bug in `get_expected_returns()`

`database_client.py:1591` uses a bare truthiness check to filter expected returns:

```python
if row['expected_return']:  # BUG: drops legitimate 0.0 values
    returns[row['ticker']] = float(row['expected_return'])
```

A user-set `0.0` expected return (e.g., for a cash position or a ticker the user wants flat) evaluates as falsy and is silently dropped. The caller then sees the ticker as "missing" and auto-generates a return, overwriting the user's explicit 0% choice.

**Fix**: Change the truthiness check to an explicit `is not None` check:

```python
if row['expected_return'] is not None:
    returns[row['ticker']] = float(row['expected_return'])
```

This is a one-line fix but it changes behavior: tickers with stored `0.0` will now correctly appear in the returns dict instead of being treated as missing.

#### 2d. Wire audit into optimization pre-flight (optional, deferred)

The optimization endpoints in `app.py` (lines ~3755-3790) already call `ensure_returns_coverage()`. We could add an `audit_expected_returns()` call that logs warnings but does not block. This is a nice-to-have for v2 -- the one-time data fix is the priority.

### Implementation steps

**Step 1: Fix DB bugs (0.0 truthiness + same-day UPSERT)**
- `database_client.py:1591`: change `if row['expected_return']:` to `if row['expected_return'] is not None:`
- `database_client.py:1759-1763`: change `ON CONFLICT (user_id, ticker, effective_date) DO NOTHING` to `ON CONFLICT (user_id, ticker, effective_date) DO UPDATE SET expected_return = EXCLUDED.expected_return, data_source = EXCLUDED.data_source`. This ensures same-day reruns overwrite stale rows instead of silently no-oping.
- Both must land before the data fix because the refresh script depends on correct save behavior

**Step 2: Data fix**
- Load `config/cash_map.yaml` to identify cash proxy tickers (`SGOV`, `IBGE.L`, `ERNS.L`)
- Exclude cash proxies from the refresh ticker list
- Run audit script to generate comparison table for remaining tickers
- Review output, confirm pipeline values are reasonable
- Write fresh values to YAML (`replace_all=True`) — this drops SGOV from expected_returns
- Save fresh values to DB for production user with `data_source='calculated'`
- Verify SGOV is NOT in the refreshed set (it should fall through to Treasury-rate path)
- Commit the YAML change

**Step 3: `audit_expected_returns()` method**
- Add to `ReturnsService` class
- Parameters: `portfolio_name`, `deviation_threshold=0.15`, `years_lookback=None`
- Returns: `{stale_tickers, audit_details, recommendation, fresh_values}`
- Must exclude cash proxy tickers from audit (they use Treasury rate, not stored values)
- Does NOT auto-fix — returns findings for caller to decide

**Step 4: `data_source` parameter on `save_expected_returns()` only**
- `DatabaseClient.save_expected_returns()`: add `data_source='user_input'` parameter, use it in the INSERT instead of hardcoded string
- Do NOT change `ensure_returns_coverage()` — auto-generated returns are intentionally temp-only
- The refresh script and `audit_expected_returns()` call `save_expected_returns(data_source='calculated')` when persisting corrected values
- No YAML path changes needed

## 5. What NOT to change

- **Do not remove YAML storage**: The YAML path is the file-mode fallback and is used in tests/dev. Keep it.
- **Do not make expected returns compute-on-demand only**: This would add FMP API calls to every optimization request. The cache-then-audit pattern is better for latency.
- **Do not add a periodic refresh cron**: Over-engineering for this use case. The audit method is sufficient.
- **Do not change the CAGR methodology**: The 10yr industry ETF CAGR method is sound. The bug is stale data, not wrong methodology.

## 9. Decision: One-time fix vs structural change

**Recommendation: One-time fix (Phase 1) + lightweight audit method (Phase 2).**

The structural alternative (compute-on-demand, never store) has significant downsides:
- Adds 5-15 seconds of FMP API latency to every optimization request
- Removes user ability to override expected returns with their own views
- Breaks the existing DB versioning model (insert-only with effective_date)

The audit method provides the safety net: if values drift stale again (e.g., portfolio composition changes but returns are not re-estimated), the audit can be run on-demand or wired into a pre-optimization check later.

## 10. Codex Review Findings (v1 FAIL → v2 → v3)

The following 5 findings from Codex review caused a FAIL verdict on v1. Each is addressed in v2:

### Finding 1: SGOV cash proxy conflict
**Issue**: SGOV is a cash proxy in `config/cash_map.yaml`. But in `get_complete_returns()` (line 403-406), stored/temp returns take precedence over the cash proxy Treasury-rate path (line 411-424). If the Phase 1 refresh script writes `SGOV=0.123`, it would shadow the correct Treasury rate.
